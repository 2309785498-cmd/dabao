# React Native App 自动化打包与分发 — 零基础也能看懂的完整指南

---

## 一、这套系统是干什么的？

简单说就是：**你在电脑上改完代码，推送到网站，手机就能自动收到新版本。**

整个过程像发快递：
1. 你打包好货物（写代码）
2. 交给快递站（推送到 GitHub）
3. 快递站自动打包发货（GitHub Actions 自动构建 APK）
4. 收货人扫码取件（手机扫码下载安装）

---

## 二、用到的工具都是什么？

先认识一下几个角色，不用管它们的技术细节，知道它们是干嘛的就行：

| 工具 | 比喻 | 实际作用 |
|------|------|----------|
| **GitHub** | 云盘 | 存代码的地方，代码改了自动同步上去 |
| **GitHub Actions** | 自动流水线 | 代码一更新，自动帮你打包成手机安装包 |
| **蒲公英** | 快递柜 | 把安装包放上去，生成一个二维码，别人扫码就能下载 |
| **React Native** | 积木框架 | 用一套代码同时生成安卓和苹果 App |
| **Expo** | React Native 的辅助工具 | 让开发更简单，还能不重新打包就更新界面 |
| **Flask** | 数据后台 | 给 App 提供数据接口，比如登录验证、数据查询 |
| **Nginx** | 门卫 | 接收 App 发来的请求，转交给后台处理 |
| **MySQL** | 仓库 | 存数据的地方，员工信息、设备信息都在这里 |

## 三、整个流程是怎么跑通的？

### 第一步：写代码 → 推到 GitHub

你在电脑上改完 App 的代码（比如改了个按钮颜色、修了个 Bug），把改动保存，然后执行 `git push` 推送到 GitHub。

推送是什么意思？就相当于把本地文件上传到网盘，只不过这个网盘会自动记录每一次改动。

### 第二步：GitHub 检测到新代码 → 自动开始打包

GitHub 里有一个叫 **Actions** 的功能，它会盯着你的代码仓库。一旦发现有新代码进来，就自动启动一台虚拟电脑，在上面执行打包任务。

这台虚拟电脑做的事情跟手动打包一模一样：
1. 把你的代码下载下来
2. 安装需要的工具（Java 环境、Node.js 环境、Android SDK）
3. 用 Gradle（安卓的打包工具）把代码编译成 APK 安装包
4. 用你提前存好的签名文件给 APK 签个名（安卓要求必须签名才能装到手机上）

整个过程大概 5 分钟。

### 第三步：APK 上传到蒲公英 → 生成二维码

打包完成后，系统自动把 APK 文件传给蒲公英的服务器。蒲公英收到文件后，会生成两个东西：
- 一个**下载链接**
- 一个**二维码图片**

这俩东西会显示在 GitHub Actions 的构建结果页面上。

### 第四步：用户扫码 → 下载 → 安装

用户用手机扫二维码，浏览器打开下载页面，点击下载 APK，安装后就能用了。

第一次安装时要允许"安装未知来源应用"（因为不是从应用商店下载的），之后更新版本不需要再设置。

## 四、两种更新方式：热更新 vs 重新打包

App 的改动分两种，处理方式不一样：

### 情况 A：只改了界面或逻辑（大部分情况）

比如改了按钮颜色、修了文字错误、调整了页面布局。这些改动不涉及 App 的底层框架。

**用热更新**：改完代码后发一个指令，App 下次打开时自动下载新版本，1 秒钟生效，用户什么都不用做。

这就像网页刷新一样——你改了网页内容，用户刷新页面就能看到新的，不需要重新下载整个浏览器。

### 情况 B：改了底层配置或装了新的功能包

比如改了安卓权限、装了新的第三方插件、改了签名配置。这些必须重新打包。

**用打 APK 的方式**：推送代码到 GitHub，自动触发打包，用户在蒲公英重新扫码下载。

### 怎么判断用哪种？

| 改动类型 | 用哪种 | 用户需要做什么 |
|----------|--------|--------------|
| 改页面样式、修 Bug | 热更新 | 什么都不用做，App 自动更新 |
| 改权限、装新插件 | 重新打包 | 扫码重新下载安装 |

## 五、后台服务器是怎么搭的？

App 不是孤立的，它需要跟服务器通信。比如登录的时候，App 把你的用户名密码发给服务器，服务器验证后返回一个"通行证"，App 拿着这个通行证才能访问其他功能。

### 服务器架构（用餐厅比喻）

```
顾客（手机 App）
    ↓ 点菜
服务员（Nginx，负责接待）
    ↓ 传菜
厨师（Flask 后台程序，处理业务逻辑）
    ↓ 取食材
冰箱（MySQL 数据库，存数据）
```

### 各角色解释：

- **Nginx（服务员）**：站在餐厅门口，接收所有 App 发来的请求，转交给后厨。比如顾客点"登录"，服务员就说"后厨，有人要登录"。

- **Flask（厨师）**：实际干活的人。验证用户名密码对不对、查员工列表、增删设备信息，都是它干的。

- **MySQL（冰箱）**：存数据的地方。员工信息、设备清单、分类目录，全在这里面。

- **Gunicorn（帮厨）**：厨师一个人忙不过来，Gunicorn 就是复制了 4 个厨师出来，同时干活，提高效率。

- **systemd（老板）**：盯着厨师，如果厨师累倒了（程序崩溃），老板立刻再开一个新厨师顶上。重不重启服务器，它都会保证服务一直在运行。

## 六、签名文件是什么？为什么要管它？

安卓手机装 App 的时候，需要验证这个 App 是不是你本人发的。签名文件就是你在这个 App 上盖的"指纹"。

如果换个签名文件，安卓会认为这是另一个 App，用户安装了会冲突。所以签名文件非常重要，必须**妥善保管，不能泄露**。

我们的做法是把签名文件加密码转成一串乱码文本，存在 GitHub 的密码箱里。打包的时候自动取出来还原成签名文件，全程不需要你手动操作。

## 七、蒲公英是什么？为什么用它？

应用商店（华为应用市场、小米应用商店）上架 App 非常麻烦：要审核、要营业执照、要软著，动辄一两周。

蒲公英是一个**内测分发平台**，专门给企业内部或小团队用，不需要审核：
- 上传 APK → 立即生成二维码
- 用户扫码 → 浏览器下载 → 安装
- 一行代码就能自动上传

适合小团队快速迭代，不用等应用商店审核。

        ↓

## 十、遇到过的坑和教训

### 坑 1：代码推上去了，但打包没触发

**原因**：打包的配置文件放错了文件夹。GitHub 只会看仓库根目录下的特定文件夹，放错了就识别不了。

**教训**：配置文件的位置很重要，必须按规矩来。

### 坑 2：打包到一半报错了

**原因**：代码仓库里少了一些必要的文件（安卓原生文件没传上去）。你在本地有这些文件，但别人（GitHub 的虚拟电脑）看不到。

**教训**：所有打包需要的文件都得传到代码仓库里，少一个都不行。

### 坑 3：手机上输入就白屏

**原因**：登录页面用了两种管理输入框的方式，它们互相冲突了。就像两个人同时抢着开同一扇门，结果门卡住了。

**教训**：一种事情用一种方式做就行，别同时用两种。

### 坑 4：手机登录失败（第一次）

**原因**：新版本安卓手机默认不允许 App 用不加密的网络传输数据（HTTP），就像外卖必须密封包装，不能敞着口。

**解决**：在安卓配置文件里加一行声明，告诉系统"这个 App 的数据不加密是可以的"。

### 坑 5：手机登录失败（第二次）

**原因**：代码里写的后台地址是开发时用的模拟器地址，不是真正的服务器地址。模拟器地址在真机上根本连不上。

**教训**：发布版本和开发版本的配置要分开，发布前检查地址是不是对的。

### 坑 6：后台服务突然访问不了

**原因**：后台程序是手动启动的，服务器重启或者程序崩了，就停了。

**解决**：用 systemd 把后台程序托管起来，就像给程序请了个保姆——程序崩了自动重启，服务器重启了也自动开机启动。
